---
title: Quarkus rest client 反序列化时，无法反序列化父类字段的问题
---

## 问题背景

当我使用Quarkus构建原生镜像，在AWS Lambda上运行时遇到了无法反序列化父类字段的问题。
调用外部接口返回的原始值包含了PredicateBase中的parameter和value,但是使用下面的对象接收时这两个字段却不存在。

Java类如下：

接口类 Predicate

```java

@JsonTypeInfo(
        use = JsonTypeInfo.Id.NAME,
        include = JsonTypeInfo.As.PROPERTY,
        property = "match"
)
@JsonSubTypes({
        @JsonSubTypes.Type(value = PredicateEquals.class, name = "EQUALS"),
        @JsonSubTypes.Type(value = PredicateRegex.class, name = "REGEX"),
        @JsonSubTypes.Type(value = PredicateBoolean.class, name = "IS"),
        @JsonSubTypes.Type(value = PredicateContains.class, name = "CONTAINS"),
        @JsonSubTypes.Type(value = PredicateNot.class, name = "NOT"),
})
@JsonIgnoreProperties(ignoreUnknown = true)
public interface Predicate {
    boolean isSatisfiedBy(final Arguments arguments);
}
```

抽象类 PredicateBase

```java
public abstract class PredicateBase implements Predicate {

    @JsonProperty
    protected Parameter parameter;

    @JsonProperty
    protected String value;
}
```

子类 PredicateEquals

```java

@Slf4j
public class PredicateEquals extends PredicateBase {

    @Override
    public boolean isSatisfiedBy(final Arguments argument) {
        log.info("Checking if satisfied with arg: {}, param: {}, value: {}", argument, this.parameter, this.value);
        return argument.with(parameter).equals(value);
    }
}
```

## 问题原因

Quarkus 构建原生镜像（Native Image） 时基于 GraalVM 编译，与传统 JVM 运行模式有本质区别：GraalVM 会在编译期做极致的静态分析和代码裁剪，
只保留能被静态追踪到的类 / 方法 / 字段，所有动态加载 / 反射的代码会被默认剔除。

而 Jackson 反序列化（尤其是多态 / 抽象父类场景）严重依赖反射：

* 解析抽象父类 PredicateBase 的 parameter/value 字段时，Jackson 需要通过反射访问字段 /getter/setter；
* 多态反序列化（根据 match 字段实例化 PredicateEquals 等子类）时，Jackson 需要反射创建子类实例；
* 若未显式配置反射规则，GraalVM 编译原生镜像时会认为这些反射代码 “不可达”，直接删除相关逻辑，最终导致：  
  ✅ 本地 JVM 测试正常 → ❌ 原生镜像中仅能解析 match（静态字段），父类 parameter/value（反射字段）为 null。

## 解决方案

### 方案一

需在 application.properties 中添加 Jackson 反射配置：

```yaml
quarkus:
  native:
    additional-build-args: >
      -H:ReflectionConfigurationFiles=reflection-config.json,
      -H:DynamicProxyConfigurationFiles=proxy-config.json,
      -H:ResourceConfigurationFiles=resources-config.json,
      -march=compatibility
```

同时，需要在src/main/resources目录下添加reflect-config.json，在json文件中配置父类/子类的反射配置

```json
[
  {
    "name": "your.package.PredicateBase",
    "fields": [
      {
        "name": "parameter"
      },
      {
        "name": "value"
      }
    ],
    "methods": [
      {
        "name": "getParameter"
      },
      {
        "name": "setParameter"
      },
      {
        "name": "getValue"
      },
      {
        "name": "setValue"
      }
    ]
  },
  {
    "name": "your.package.PredicateEquals",
    "methods": [
      {
        "name": "isSatisfiedBy"
      }
    ]
  }
]
```

### 方案二

在Java类上添加注解 @RegisterForReflection
PredicateBase

```java

@RegisterForReflection
public abstract class PredicateBase implements Predicate {

    @JsonProperty
    protected Parameter parameter;

    @JsonProperty
    protected String value;
}
```

PredicateEquals

```java

@Slf4j
@RegisterForReflection
public class PredicateEquals extends PredicateBase {

    @Override
    public boolean isSatisfiedBy(final Arguments argument) {
        log.info("Checking if satisfied with arg: {}, param: {}, value: {}", argument, this.parameter, this.value);
        return argument.with(parameter).equals(value);
    }
}
```

## 为什么本地 JVM 测试不需要这些配置？

| 运行环境         | 类加载 / 反射规则                              |
|--------------|-----------------------------------------|
| 本地 JVM       | 动态加载类，反射无需提前声明，Jackson 可自由访问所有字段 / 方法   |
| Quarkus 原生镜像 | GraalVM 静态编译，仅保留静态可追踪的代码，反射代码必须显式声明才能保留 |

## 总结

1. 核心作用：解决 GraalVM 静态编译对 Jackson 反射的裁剪问题，确保原生镜像中能正常反序列化抽象父类的字段；
2. 关键价值：
    - 保留 Jackson 核心类的初始化能力；
    - 保留 PredicateBase 字段 /getter/setter 的反射访问权；
    - 保留多态子类（如 PredicateEquals）的实例化和方法调用权；
